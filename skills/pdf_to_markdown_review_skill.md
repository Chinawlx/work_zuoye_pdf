# Skill: PDF → Markdown 解析质量检查与人工复核提示

## 1. Skill 名称与适用场景

**名称**: `pdf_to_markdown_review`

**目标**: 当智能体拿到「一份 PDF + 它的 Markdown 转换结果 + 结构化中间产物(blocks/tables)」时,先自动跑一轮解析质量检查,产出问题清单、风险标记和人工复核清单,让人工不必从零逐页对照,只需聚焦在机器无法判定的少数点上。

**适用场景**:
- 财报/附注、研报、合同、招股书等以「标题 + 正文 + 表格」为主的结构化文档。
- 已有一套 PDF→Markdown 转换流程(如本仓库 `convert_pdf.py`),需要在交付给 RAG / 检索 / 人工之前做质量门禁。
- 批量转换后,需要快速筛出「哪些页/哪些表有风险」以分配人工。

## 2. 不适用场景 / 必须人工介入的场景

- **纯扫描件且无 OCR 结果**: 没有任何文字层也没跑 OCR 时,本 Skill 无内容可检查,直接判 `BLOCKED`,要求先补 OCR。
- **关键财务/法律数值的最终采信**: 本 Skill 只标风险,不替人拍板。涉及对外披露、入账、合同金额的数字,必须人工签字。
- **扫描页(OCR/VLM 来源)的表格数值**: VLM OCR 无逐字置信度且可能漏行/串列,**一律 NEED_RECHECK**,不得自动通过。
- **跨页表格、合并单元格、嵌套表头**: 结构判断不可靠时转人工。
- **PDF 与外部 GT(如 Excel)对账**: 两源不一致时只标 `NEED_RECHECK`,不自动补数。

## 3. 输入文件要求

| 输入 | 是否必需 | 说明 |
| --- | --- | --- |
| 原始 PDF | 必需 | 用于回溯来源、必要时重渲染某页 |
| `document.md` | 必需 | 转换后的 Markdown,需含 `<!-- page: N -->` 页码标记 |
| `blocks.json` | 必需 | 逐页文本块,含 `type` / `bbox`(source locator) / `is_ocr` |
| `tables.json` | 必需 | 表格结构: 表标题/表头/行列/`period_columns`/`total_check`/`is_ocr` |
| 解析日志 / `qa_report.md` | 可选 | 已有自动报告时可增量复核 |

若缺 `blocks.json` 或 `tables.json`,降级为「只能查页码和表格渲染」,并在报告中注明检查不完整。

## 4. 输出格式

输出一份 `review_report.md`,固定四块:

1. **质量概览**: 总页数、文字页/扫描页数、表格数、自动通过/存疑/阻断计数。
2. **问题清单**: 每条含 `[级别] 位置(page/table_id/bbox) — 问题描述`。级别分 `BLOCK / WARN / INFO`。
3. **需人工复核清单**: 机器无法判定、必须人看的点,每条给出复核动作建议。
4. **优化建议**: 针对本次发现,建议如何改 prompt / 调阈值 / 补规则。

## 5. 执行步骤

> 顺序固定,从「能否追溯」到「内容对不对」逐层推进。

1. **页码检查**: 遍历 `document.md` 的 `<!-- page: N -->`,确认连续不缺页;与 `blocks.json` 里每页 `printed_page_no`(页内打印页码)交叉核对,不一致标 WARN。
2. **标题层级**: 检查每页是否有且仅有合理的 H1;H2(表标题)是否紧邻其表格;有无正文被误判成标题或反之(看 `font_size` 与 `type`)。
3. **正文完整性**: 抽查段落是否被截断、页眉页脚("第 N 页""XX公司…节选")是否被正确剔除而非混入正文。
4. **表格检查**: 对每个 table 核对表头非空、列数一致(各行 `len(row)` 相等)、有无空行名、`period_columns` 是否识别到。
5. **图片页 / 扫描页**: 凡 `is_ocr=true` 的 block/table,确认 `document.md` 中有显式扫描页标记与 NEED_RECHECK 提示;这些页的数值不计入自动通过。
6. **来源追溯**: 抽查每个 block/table 是否带 `page` + `bbox`;扫描页至少能定位到整页。来源缺失标 WARN。
7. **数值检查**: 读 `total_check`——`pass` 记通过、`fail` 升级为 BLOCK、`NEED_RECHECK`(OCR 表)进人工清单。

## 6. 质量检查 checklist

| 检查项 | 可自动判定 | 通过标准 |
| --- | --- | --- |
| 页码连续 | ✅ | `<!-- page: N -->` 1..N 无缺 |
| 打印页码一致 | ✅ | `printed_page_no == 物理页序` |
| 标题层级 | 半自动 | 每页 1 个 H1;表标题为 H2 且贴近表格 |
| 表格完整性 | ✅ | 表头非空、无空行名 |
| 行列结构 | ✅ | 每行列数一致 |
| 数值/合计 | ✅(文字页) | 明细之和 == 合计(容差 0.01) |
| 单位 | 半自动 | 表注声明的单位已记录(如「百万元」) |
| 脚注/注释 | 半自动 | `注/补充/脚注` 正确归为 note,未混入表格 |
| 期间列 | ✅ | 识别到形如 `2026-06-30` 的列 |
| 扫描页标记 | ✅ | `is_ocr=true` 页有 needs_ocr 提示 |
| source locator | ✅ | 每 block/table 带 page+bbox |
| OCR 数值 | ❌ 仅人工 | 一律 NEED_RECHECK |

## 7. 常见错误与处理方式

| 现象 | 处理 |
| --- | --- |
| **页眉污染**: "XX公司…节选"混入正文 | 按 y0 位置(顶部/底部)识别并剔除,降级为 header/footer,不进正文 |
| **表格错位/串列** | 核对各行列数;不一致标 BLOCK 转人工 |
| **跨页表格** | 按表头连续性合并;无法确定则两段都保留并标 WARN |
| **OCR 低质/漏行** | OCR 表强制 NEED_RECHECK;提示重渲染提高 DPI 或换 prompt |
| **VLM 表格格式漂移** | 三种格式归一: ①标准 ②缺首尾竖线 `a\|b\|c` ③散装每格一行。归一失败则保留原文不臆造 |
| **乱码/编码错** | 检查字体是否内嵌(Identity-H 无 CIDToGID 易乱码),建议改用图片+视觉识别该页 |
| **空页** | 标 INFO,确认非漏抽 |
| **图片说明缺失** | 图片页无 caption 时标 WARN,提示人工补 |
| **合计≠明细** | BLOCK,必须人工核对(可能是漏行或 OCR 错) |

## 8. 人工复核点(机器不能自动判断)

- 扫描页/OCR 来源的**任何关键数值**(漏行、串列、单字错认机器看不出)。
- **跨口径行名**: 如「工资、奖金、津贴和补贴」这类含顿号的宽口径项,在 Excel 中可能被拆成多行,是否同一口径需人判断。
- **PDF 与外部 GT 不一致**: 只标 `NEED_RECHECK`,补不补由人决定。
- **合并单元格 / 嵌套表头**的语义归属。
- **单位与符号**: 百万/千元、百分比、负数括号写法等口径。

## 9. 如何持续优化(bad case 回写闭环)

1. 人工复核发现新错误类型 → 记成一条 bad case(输入片段 + 期望输出 + 实际输出)。
2. 归类: 属于「prompt 没说清」→ 改 OCR_PROMPT;属于「归一规则没覆盖」→ 在表格归一/checklist 加规则;属于「阈值不合适」→ 调 `HEADER_MARGIN`/`H1_FONT_SIZE` 等。
3. 把该 bad case 固化成一条回归用例,纳入测试,确保下次不复发。
4. 周期性统计各类问题占比,优先治理高频项——让人工每次只需看更少的点。

> 原则: 每修一个 bad case,就把「本来要人看的一类问题」变成「机器能自动标」,人工工作量单调下降,但**永不归零**(OCR 数值、跨口径、对账分歧始终留给人)。

## 10. 小示例

**输入**(一段 Markdown + 一条表格解析结果):

```md
<!-- page: 3 -->
> 🖼️ 扫描页 / 无文字层 — OCR(VLM) 生成, needs_ocr: true
表38-1 预计负债变动
| 项目 | 期初余额 | 期末余额 |
| 产品质量保证 | 15.00 | 18.20 |
| 合计 | 23.50 | 26.00 |
```
```json
{"table_id":"p3_ocr_t1","is_ocr":true,"header":["项目","期初余额","期末余额"],
 "rows":[["产品质量保证","15.00","18.20"],["合计","23.50","26.00"]],
 "total_check":{"status":"NEED_RECHECK"}}
```

**输出**(质量判断 + 复核建议):

- `[WARN] page3/p3_ocr_t1` — 表格缺少表头分隔行 `| --- | --- | --- |`,部分渲染器不显示为表格,建议归一时补上。
- `[WARN] page3/p3_ocr_t1` — 表头声明 3 列,但「合计」行明细疑似漏行:仅「产品质量保证」一条明细,合计 23.50 ≠ 15.00,**可能漏了一行**。
- `[BLOCK→人工] page3` — `is_ocr=true`,数值 15.00/18.20/23.50/26.00 全部 NEED_RECHECK,需对照原 PDF 第 3 页人工逐项核对,确认是否漏行。
- `[INFO]` source locator 完整(page=3,扫描页定位到整页),可追溯。

> 判断结论: **不通过自动门禁**,转人工。复核动作: 打开原 PDF 第 3 页,核对预计负债表是否还有「未决诉讼」等明细行被漏,并校验合计。
